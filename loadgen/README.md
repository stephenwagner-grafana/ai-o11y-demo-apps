# loadgen

The central synthetic-traffic generator for the AI o11y demo. One pod in
the `k6-loadgen` namespace drives **both** apps (NeonCart + SupportBot) —
there is no per-app loadgen.

The loadgen produces ~3-5 LLM calls/minute total: realistic-looking but
not a stress test. Its job is to keep the dashboards populated with the
kind of varied AI-shaped activity a small e-commerce + support-chat
business would produce on an average day.

For the full design contract (cohort sizes, journey weights, env vars,
gateway throttle semantics), see [docs/LOADGEN.md](../docs/LOADGEN.md).

## How it works

```
            ┌─────────────────────────────┐
            │  loadgen-orchestrator (py)  │
            │  FastAPI on :8080           │
            │  /health, /readyz, /metrics │
            └──────────────┬──────────────┘
                           │ supervises (spawn/SIGTERM)
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
       k6 process     k6 process     k6 process    …
       nc-non-ai      nc-gift        nc-chat       sb
            │              │              │             │
            └─ HTTPS ──────┴──────────────┴─────────────┘
                           ▼
                   nc-web / sb-web
                           ▼
                       llm-gateway
                           ▼
                  Anthropic / OpenAI / Gemini / Ollama
```

- **Orchestrator** (`orchestrator/main.py`) is a small FastAPI service.
  It reads the synthetic user pool from `users.yaml`, polls the LLM
  gateway's `/open` endpoint every 5 s, and starts/stops one K6
  subprocess per scenario in response to provider state.
- **K6 scenarios** (`k6/scripts/*.js`) do all the actual HTTP work
  against NeonCart and SupportBot. The orchestrator passes each scenario
  its slice of the user pool via a temp JSON file (`USERS_FILE` env).

The orchestrator does NOT call LLM endpoints directly. All LLM-bearing
HTTP goes through K6 scripts, which in turn hit the public web apis on
`nc-web` and `sb-web`.

## Why this shape?

- **K6 owns the request work.** K6 has the right primitives for
  multi-VU, weighted journeys, ramping/arrival-rate executors, and
  iterates cheaply in JS. Python would be a worse fit.
- **Python owns the supervision.** We need to react to the gateway's
  `/open` state in real time and tear down individual scenarios without
  killing the others (the 150 non-AI NC users keep shopping when Claude
  closes). That coordination is much easier in a small async Python
  service than in JavaScript inside K6.

## Gateway coordination

Loadgen polls `${GATEWAY_URL}/open` every
`LOADGEN_POLL_OPEN_INTERVAL_SEC` seconds (default 5). The poller watches
`providers.anthropic` specifically — Claude is the rate-defining
provider per the design doc. **NOT** `any_open`.

| Anthropic state | NC non-AI K6 | NC gift-finder / chatbot / both K6 | SB K6 |
|---|---|---|---|
| open    | running | running | running |
| closed  | running | SIGTERM'd, restarted on reopen | SIGTERM'd, restarted on reopen |

In-flight K6 iterations get `SIGTERM` and ~30 s to drain (their current
iteration finishes), then `SIGKILL` if still alive. New K6 spawns when
the gateway transitions back to open.

## Critical: every request carries `X-Caller-Type: synthetic`

The K6 scripts set `X-Caller-Type: synthetic` on **every** HTTP request
via the helper in `_common.js`. The gateway routes:

- `synthetic` → random across configured + open providers, respects
  `/open`, counts against caps
- (no header / `interactive`) → always Anthropic, ungated

If the header is missing, loadgen traffic gets treated as a real human
and pinned to Claude — defeating the multi-provider story. The header is
checked into `_common.js`'s `syntheticHeaders()` and used by `request()`
so individual scripts physically can't forget it.

## User pool

`config/users.yaml` is the runtime source of truth — 200 NC + ~30 SB
users with **stable cohort assignment** (a "gift-finder" user always
uses the gift-finder, never the chatbot). See
`config/users.yaml.example` for the schema.

The real file is generated by `tools/regenerate-users.py`
(separately scaffolded). Hand-edits survive loadgen restarts; only
re-running the script overwrites them.

The orchestrator reads `USERS_CONFIG_PATH` (default
`/etc/loadgen/users.yaml`) once at startup. Restarting the orchestrator
re-reads the file — there's no hot reload.

## Environment variables

| Var | Default | Notes |
|---|---|---|
| `NC_TOTAL_USERS` | 200 | Informational; real count comes from users.yaml |
| `NC_AI_ADOPTION_RATE` | 0.25 | Informational |
| `NC_SESSIONS_PER_HOUR` | 60 | Target session arrival rate across all NC cohorts |
| `SB_TOTAL_USERS` | 30 | Informational |
| `SB_SESSIONS_PER_HOUR` | 30 | Target session arrival rate for SB |
| `LOADGEN_TIME_OF_DAY` | `false` | If true, 1.5× rate 9-5 in LOADGEN_TZ, 0.5× at night |
| `LOADGEN_TZ` | `America/New_York` | TZ for the time-of-day curve |
| `LOADGEN_POLL_OPEN_INTERVAL_SEC` | 5 | How often to poll the gateway |
| `LOADGEN_NC_BASE_URL` | `http://nc-web.neoncart.svc.cluster.local` | Override for cluster-external testing |
| `LOADGEN_SB_BASE_URL` | `http://sb-web.support-bot.svc.cluster.local` | Override for cluster-external testing |
| `SB_USER_DOMAIN` | `acme.com` | Email domain for SB synthetic users |
| `USERS_CONFIG_PATH` | `/etc/loadgen/users.yaml` | Where to read the user pool |
| `GATEWAY_URL` | `http://llm-gateway.llm-gateway.svc.cluster.local` | Loadgen polls `${GATEWAY_URL}/open` |
| `K6_BINARY` | `k6` | Override if you ship a custom build |
| `LOADGEN_SCRIPTS_DIR` | `/loadgen/k6/scripts` | Where the .js files live in the image |

## Endpoints (orchestrator)

| Method | Path | Notes |
|---|---|---|
| `GET` | `/health` | Liveness for k8s |
| `GET` | `/readyz` | Readiness — 503 until the orchestrator has started |
| `GET` | `/status` | JSON: per-scenario running state + user count + gateway state |
| `GET` | `/metrics` | Prometheus: `loadgen_gateway_anthropic_open`, `loadgen_k6_processes_running{scenario=…}`, `loadgen_k6_restarts_total{scenario=…}`, `loadgen_k6_terminations_total{scenario=…}`, `loadgen_gateway_poll_failures_total` |

The orchestrator's `/metrics` is intentionally narrow: only loadgen
internal accounting. The interesting metrics (LLM cost, token usage,
journey outcomes) come from the gateway and the apps themselves — those
are what dashboards consume.

## Local dev

```bash
# Spin up K6 + Python with a fake users.yaml and a stubbed gateway:
pip install -r requirements.txt

# Run a single scenario against a local nc-web (skipping the orchestrator)
k6 run \
  --env USERS_FILE=$(pwd)/config/users.yaml.example \
  --env NC_BASE_URL=http://localhost:8000 \
  --env SESSIONS_PER_HOUR=60 \
  k6/scripts/neoncart-non-ai.js

# Run the orchestrator (will spawn k6 subprocesses; needs k6 on PATH)
USERS_CONFIG_PATH=$(pwd)/config/users.yaml.example \
GATEWAY_URL=http://localhost:8000 \
LOADGEN_NC_BASE_URL=http://localhost:8000 \
LOADGEN_SB_BASE_URL=http://localhost:8001 \
LOADGEN_SCRIPTS_DIR=$(pwd)/k6/scripts \
uvicorn orchestrator.main:app --host 0.0.0.0 --port 8080
```

## Docker

```bash
docker build -t ai-o11y-loadgen .
docker run --rm -p 8080:8080 \
  -v $(pwd)/config/users.yaml.example:/etc/loadgen/users.yaml \
  -e GATEWAY_URL=http://host.docker.internal:8000 \
  -e LOADGEN_NC_BASE_URL=http://host.docker.internal:8000 \
  -e LOADGEN_SB_BASE_URL=http://host.docker.internal:8001 \
  ai-o11y-loadgen
```

## Scope

In scope for the base loadgen:

- 200 NC + 30 SB users, 75/25 AI split, 60/30/10 gift-finder/chatbot/both
- Weighted journeys per cohort (see docs/LOADGEN.md)
- Occasional "show me mice" prompt (~1.5% of nc-chatbot iterations) to
  keep the signature demo trace populated
- Synthetic-vs-interactive header on every request
- Gateway throttle response (`/open`-driven scenario start/stop)

Out of scope (extensions add these later):

- Scripted bad-actor traffic (Tim-as-data-thief, etc.)
- Outlier spikes and burst patterns
- Prompt-injection adversarial traffic
- Domain-specific bursts (Black Friday, etc.)
